# Design: `.run-site-env.sh` — sourceable env export

**Date:** 2026-06-19
**Status:** Approved (pending spec review)

## Problem

When `run-site` is running a dev stack, the running Postgres/Redis endpoints
and Django config (secret key, allowed hosts) live only inside the in-memory
environment that `build_subprocess_env()` injects into each subprocess
(`migrate`, `runserver`, `celery`). A developer in a **second terminal** has no
easy way to talk to the *same* running services — to run `manage.py dbshell`,
`manage.py shell`, or a bare `psql` against the live container.

There is already a consolidated `.run-site-config` sidecar (TOML), but TOML is
awkward to consume from a shell: you can't `source` it, and `psql` won't read
it. The user wants a single file they can `source` in a second terminal so that
both `manage.py` and native PG tools "just work" against the running stack.

(Historical note: the user vaguely recalled "a separate dotfile per port".
That mechanism belonged to the older `django-dev-helpers` companion package
— `.dev_helpers_token`, `.dev_helpers_pg_port`, etc. This `run-site` package
already superseded it with the single `.run-site-config` sidecar. The new
export file described here is an additional, shell-friendly artifact, not a
return to per-port files.)

## Goal

After services start, `run-site` writes a single **sourceable shell file**,
`<project_root>/.run-site-env.sh`, containing `export` statements for the
project-facing configuration. A developer can:

```sh
source .run-site-env.sh
python manage.py dbshell      # uses DATABASE_URL / DJANGO_* 
psql                          # uses PGHOST/PGPORT/PGDATABASE/PGUSER/PGPASSWORD
```

## Non-goals

- **No architectural inversion.** `run-site` continues to build and inject its
  subprocess environment in-memory exactly as it does today. The
  `.run-site-env.sh` file is an *additional output artifact*; `run-site` never
  reads it back to launch its own processes. (Round-tripping through a file
  would add stale-file / parse-drift failure modes for no benefit.)
- No new opt-in flag. The file is always written, mirroring the existing
  `.run-site-config` sidecar lifecycle.
- No new secret-exposure surface: `.run-site-config` already records the PG
  password in plaintext today, so this introduces no new class of leak. It is
  gitignored (see below).

## File contents

The file is generated from the same data `_project_values()` already computes
for the sidecar and subprocess env — **single source of truth, no parallel
logic.** Full-stack example (Postgres + Redis, non-loopback bind):

```sh
# Auto-generated by run-site. Recreated on each start, removed on clean
# shutdown. Contains secrets — keep `.run-site-env.sh` out of version control.
#
# Usage:  source .run-site-env.sh
export DATABASE_URL="postgres://user:pass@127.0.0.1:55432/dbname"
export DJANGO_SECRET_KEY="..."
export DJANGO_ALLOWED_HOSTS="localhost,127.0.0.1,myhost.local"
export REDIS_URL="redis://127.0.0.1:56379/0"
export PGHOST="127.0.0.1"
export PGPORT="55432"
export PGDATABASE="dbname"
export PGUSER="user"
export PGPASSWORD="..."
```

### Variable set

| Variable | Source | Emitted when |
|----------|--------|--------------|
| `DATABASE_URL` | `_project_values()["database_url"]`, honoring `[env].database_url` name | Postgres **or** SQLite enabled |
| `DJANGO_SECRET_KEY` | persisted secret, honoring `[env].secret_key` name | a secret key is in play (`secret_key is not None`) |
| `DJANGO_ALLOWED_HOSTS` | `compute_allowed_hosts()`, honoring `[env].allowed_hosts` name | non-loopback bind (list is non-empty) |
| `REDIS_URL` | `_project_values()["redis_url"]`, honoring `[env].redis_url` name | Redis enabled |
| `PGHOST` `PGPORT` `PGDATABASE` `PGUSER` `PGPASSWORD` | endpoints + `config.postgres` | Postgres enabled (omitted entirely in SQLite mode) |

Notes:
- The URL/Django vars respect the project's `[env]` mapping names (so a project
  that renamed `DATABASE_URL` gets its chosen name), exactly like the in-memory
  env. The `PG*` names are **fixed** libpq conventions and are not remappable.
- Disabled services omit their lines entirely (no Redis → no `REDIS_URL`; SQLite
  mode → no `PG*`, and `DATABASE_URL` carries the `sqlite:///…` form).
- The `DEV_HELPERS_*` contract and the inherited `os.environ` are **not**
  included — this file is the clean project-facing subset.

### Format

- **`export KEY="value"`** form. Rationale: the primary use is `source`-ing in a
  shell, which requires the vars to be exported to child processes; `export`
  gives that directly. The `export ` prefix is also stripped by `python-dotenv`
  and `django-environ`, so the file additionally works for dotenv loaders.
- Values are **double-quoted with escaping** so passwords/secrets containing
  `$`, `` ` ``, `"`, `\`, spaces, or newlines survive a `source` intact. A
  dedicated `_sh_double_quote()` helper escapes `\`, `"`, `` ` ``, and `$`
  (the four characters special inside bash double quotes) and is unit-tested
  against adversarial values.

## Code structure

New module `src/run_site/env_file.py`, mirroring `sidecar.py`:

```python
ENV_FILENAME = ".run-site-env.sh"

@dataclass(frozen=True)
class EnvFileInfo:
    # the resolved KEY -> value pairs to export, already name-mapped
    exports: tuple[tuple[str, str], ...]

def env_file_path(project_root: Path) -> Path: ...
def write_env_file(*, project_root: Path, info: EnvFileInfo) -> Path: ...
def remove_env_file(*, project_root: Path) -> None: ...   # best-effort, never raises
def _render(info: EnvFileInfo) -> str: ...
def _sh_double_quote(value: str) -> str: ...
```

A small builder assembles `EnvFileInfo.exports` from `RunSiteConfig` +
`ContainerEndpoints` + the resolved env mapping. To avoid duplicating the
service-enabled/disabled logic, it reuses `_project_values()` and
`effective_env_mapping()` from `env.py` (lifting `_project_values` to a public
name if needed) for the URL/Django vars, and reads the `PG*` parts from
`config.postgres` + `endpoints` directly.

### Wiring in `cli.py`

- After services start, call `write_env_file(...)` immediately adjacent to the
  existing `write_sidecar(...)` call, fed from the same `config` + `endpoints`
  + `secret_key` + `lan_hosts` already in scope.
- In the shutdown path, call `remove_env_file(...)` adjacent to
  `remove_sidecar(...)`.

## `.gitignore` fix (drive-by, in scope)

The current gitignore pattern is `.run_site_*` (underscores) but the actual
sidecar is `.run-site-config` (hyphens) — they do **not** match, so the sidecar
is not currently ignored. Replace the underscore pattern with a hyphen glob
that covers the whole family:

```
# django-run-site dotfiles
.run-site-*
```

This now covers `.run-site-config` (fixing the existing miss) and the new
`.run-site-env.sh`. Keep the legacy `.dev_helpers_*` lines for the companion
package. Apply the same correction to the `.gitignore` snippet that the `init`
command suggests/writes (in `init_cmd.py`), if it carries the broken pattern.

## Testing

`tests/test_env_file.py`, pure rendering (no Docker), mirroring
`tests/test_sidecar.py`:

1. Full stack (PG + Redis, non-loopback bind) → all vars present, correct values.
2. Redis disabled → no `REDIS_URL`.
3. SQLite mode → `DATABASE_URL` is `sqlite:///…`, no `PG*` lines.
4. Loopback bind → no `DJANGO_ALLOWED_HOSTS`.
5. `[env]` mapping rename → URL var uses the custom name; `PG*` stay fixed.
6. Quoting: password containing `"`, `$`, `` ` ``, `\`, space → `source`-safe
   output (assert the escaped string round-trips through a shell parse).
7. `remove_env_file` is idempotent / never raises when the file is absent.
8. `write_env_file` overwrites a stale file from a crashed run.
